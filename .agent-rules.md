# Agent Rules — Visual Feedback Plugin

## GitHub Push Protocol

Whenever you push commits to GitHub for this project (`visual-feedback-plugin`), you **MUST** follow this protocol:

### Step 1 — Always ask before pushing
Before running `git push`, pause and ask the user:

> "Should this push also be a **versioned release** on GitHub?
> - **Yes** → I'll bump the version, tag it, and create a GitHub Release so WordPress sites get the update notification.
> - **No** → I'll just push the commits as usual (no version bump, no release)."

Wait for the user's explicit answer before proceeding.

---

### Step 2a — If YES (version release)

1. **Ask which version bump** if not obvious from the changes:
   - `patch` (bug fixes, copy tweaks) → e.g. `0.2.0` → `0.2.1`
   - `minor` (new features, backwards-compatible) → e.g. `0.2.0` → `0.3.0`
   - `major` (breaking changes) → e.g. `0.2.0` → `1.0.0`

2. **Update the version string** in two places:
   - `wordpress-plugin/agency-feedback-wp/agency-feedback-wp.php` — the `Version:` header AND the `AFWP_VERSION` constant
   - Confirm the new version number with the user before committing

3. **Commit the version bump** with message:
   ```
   chore: bump version to vX.X.X
   ```

4. **Push commits** to `main`.

5. **Create a GitHub Release** using the GitHub MCP tool:
   - Tag: `vX.X.X`
   - Title: `vX.X.X`
   - Body: summarise what changed in this release (based on recent commits)
   - Mark as **latest release**

6. Confirm to the user:
   > "✅ Released as `vX.X.X` on GitHub. WordPress sites will see the update notification within 12 hours."

---

### Step 2b — If NO (just push)

1. Run `git push` normally.
2. Remind the user:
   > "Pushed. No release created — WordPress sites won't see an update for this push."

---

## General Rules

- **Never** create a GitHub Release without the user's explicit approval.
- **Never** push a version bump commit unless a release was agreed upon.
- The version in `agency-feedback-wp.php` (`Version:` header + `AFWP_VERSION`) must **always match** the GitHub Release tag (without the `v` prefix).
- GitHub Release tags must always be in the format `vX.X.X` (semver with a `v` prefix).
- The GitHub repo for this project is: `lynesslim/visual-feedback-plugin`
